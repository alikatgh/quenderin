# r13 — Quenderin i18n audit (2026-07-11)

**Lens:** Hardcoded strings, plural rules, locale loaders (50-round plan r13)
**Verdict: deliberate N/A for externalization; two locale-behavior notes logged.**

## Status
The product is **English-only by design** at this stage (all three clients). There is no i18n
framework, no locale loader, and every user-facing string is hardcoded English — consistently, on
purpose. Introducing extraction infrastructure now would be speculative scaffolding; revisit when
a second locale is actually planned (that project should start from the store-compliance audit's
market list).

## Locale-behavior notes (not bugs today, will bite any future localization)

1. **Dot-decimal only in tool parsers.** `parseConversion` (`src/services/tools/unitConvert.ts:99`)
   and the calculator tokenizer accept only `[0-9.]` — a comma-decimal input ("3,5 km to mi")
   degrades to the hint message. This is PARITY-PINNED behavior (the Swift/Kotlin twins scan
   decimal digits identically; see `tokenizer-decimal-digit-only` vector) — any future locale work
   must change all three twins together, not just the desktop.
2. **Browser-locale formatting is already correct where it matters.** UI timestamps use
   `toLocaleString`-family APIs (respect the OS locale); numbers that must be stable across
   platforms (calculator observations) deliberately do NOT use locale formatting — that's the
   canonical `NumberRender` contract, keep it locale-independent forever.
